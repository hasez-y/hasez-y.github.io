# 持久化

Redis 用作缓存之所以快，是因为在内存操作，但是内存会因为系统故障，断电等导致 Redis 数据丢失，所以需要一种持久化机制来保存内存数据到硬盘。

Redis 有两种持久化方法

  1. RDB 快照
  2. AOF 追加文件

# RDB

在指定时间间隔内将内存中的数据集快照写入硬盘，需要恢复数据时直接读取快照到内存即可。

  • 适合大规模数据恢复，且对数据完整性和一致性要求不高
  • 由于在一个间隔时间内保存一次快照，所以 Redis 宕机后，会丢失最后一次快照的修改

# AOF

以日记的形式记录每个写操作，将 Redis 执行过的所有写指令记录下来，只允许追加文件但不可以改写文件，Redis 启动初会读取文件重新构建数据。

AOF 采用文件追加方式，文件会越来越大，为了避免该清空，新增重写机制，当 AOF 文件的大小超过指定阈值，Redis 就会启动 AOF 文件的内容压缩，只保留可以恢复数据的最小指令集。

AOF 有三种模式

  1. appendfsync always 同步持久化，每次数据更改都会记录到磁盘
  2. appendfsync everysec 异步操作，每秒同步到磁盘
  3. appendfsync no 从不同步

AOF 文件要远大于 RDB 文件，恢复速度慢于 RDB

如果是用内存作为数据库，那么建议 RDB 和 AOF 都开启

# 过期键的删除策略

Redis 过期删除策略是 惰性删除和定期删除两种配合使用

  1. 惰性删除：不会主动删除过期 Key，只有访问该数据时，再检查是否过期，如果过期则删除并返回 null
  2. 定期删除：Redis 会周期性的随机测试一批设置了过期时间的 key 并进行处理。

# Redis 内存淘汰策略

Redis 是不断的删除一些过期数据，但是没有过期的 key 也会越来越多，那么 Redis 内存不够用时就要使用内存淘汰机制了。

当 Redis 的内存超过最大允许的内存之后，会触发淘汰策略，删除一些不常用的数据，以保证 Redis 的正常运行。

内存淘汰机制有很多种，这里只说最常用的 allkeys-lru，当内存不足以容纳新写入数据时，在键空间中，移除最近最少使用的 key

内存淘汰策略可以通过配置文件来修改
