# InnoDB 索引

众所周知 MySQL 中的 InnoDB 索引结构采用 B + 树，如果了解从二叉树到 B+ 树的演变，那么可能认为 B + 树 更适合做文件存储，所以 InnoDB 采用了 B + 树，而不是 B 树。

# 那 MongoDB 为什么采用 B 树呢？

在此之前我们先对比 B + 树 与 B 树 区别，就能知道 MongoDB 为什么采用 B 树了。

B 树的查询效率并不稳定，不像 B + 树一样将数据存储在叶子节点，但是因此 B 树单一查询平均性能要比 B + 树要好，但是区间查找时或遍历树时，就比较耗时，因为要遍历局部树或整棵树，因此 B + 树在 B 树基础上在叶子节点添加了指针方便区间查找和遍历。

知道以上区别，那么得出 MongoDB 为什么采用 B 树：B 树查找单个数据时平均性能优于 B + 树，MongDB 采用 B 树原因就是不经常使用遍历与区间查询。

那么又会有疑问，为什么 MongoDB 不经常遍历或区间查询呢？

那是因为 MongoDB 的数据存储像 JSON 一样，不像关系型数据库一样；假设有一对多的查询，关系型数据库有两个表，学生表和班级表，那么查询某个班级有多少人时，那边关系型数据库必定要区间遍历学生表，那么 B + 树的叶子节点指针就会发挥作用。
