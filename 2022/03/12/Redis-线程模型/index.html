# Redis 是多线程还是单线程

Redis 6.0 之前，Redis 的核心网络模型选择用单线程来实现，而之后在网络模型中实现多线程 I/O，注意这里提到的都是网络模型线程，而 Redis 执行命令依旧是单线程

Redis 的版本有两个重要的节点：

  1. Redis 4.0 引入多线程处理异步任务
  2. Redis 6.0 在网络模型中实现多线程

网络上说 Redis 是单线程，通常指 Redis 6.0 之前，其核心网络模型使用的是单线程

而 6.0 后引入多线程 I/O，只是用来处理网络数据的读取和协议解析，而执行命令依旧是单线程。

这么说 Redis 一定全是单线程执行任务吗？不是，Redis 4.0 时引入了多线程来处理异步操作，这是针对那些非常耗时的命令，避免堵塞单线程。

# Redis 的瓶颈

对于一个 DB 来说，CPU 通常不会是瓶颈，而是 I/O；而 Redis 排除了持久化方案外，是完全的纯内存操作，没有 I/O 的耗时操作所以非常快

那 Redis 的瓶颈在哪？答案在网络 I/O，也就是客户端和服务端之间的网络传输延迟，因此 Redis 选择了 单线程的 I/O 多路复用来实现它的核心网络模型。

# Redis 6.0 多线程

随着互联网业务系统处理的线上流量越来越大，Redis 的单线程模式导致系统消耗很多 CPU 时间在网络 I/O 上浪费性能。

所以在实现核心网络模型上使用了多线程来提高系统吞吐量。

采用多线程后，性能的提升至少是一倍以上。

# Redis 6.0 开启多线程后，是否存在线程并发安全问题？

从实现机制可以看出，Redis 的多线程部分只是用来处理网络数据的读写和协议解析，而执行命令仍是单线程顺序执行，所以不需要考虑线程安全问题。
