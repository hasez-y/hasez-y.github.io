# ClassLoader 类加载器

了解双亲委派机制前，需要知道 JVM 的 ClassLoader 类加载器

总所周知 Java 是运行在 Java 的虚拟机 (JVM) 中的

  1. 我们在 IDE 编写好 Java 源代码
  2. 将源代码编译成 .class 字节码文件
  3. 最终由 JVM 的 ClassLoader 将这些 .class 字节码文件加载到 JVM 中

# JVM 提供了三层的 ClassLoader

  • Bootstrap classLoader 启动类加载器

    由 C++ 编写，加载 java 核心库 java.* ，构造 ExtClassLoader 和 AppClassLoader ，由于引导类加载器涉及到虚拟机本地实现细节，开发者无法直接获取到启动类加载器的引用，所以不允许直接通过引用进行操作。

  • ExtClassLoader 标准扩展类加载器

    由 Java 编写，加载扩展库，如 classpath 中的 jre ， javax.* 或者
    java.ext.dir 指定位置中的类，开发者可以直接使用标准扩展类加载器。

  • AppClassLoader 系统类加载器

    由 Java 编写，加载程序所在的目录。

我们写的类都是由 AppClassLoader 系统类加载器加载的；也可以自定义类加载 CustomClassLoader 来加载指定路径的 class 文件

# 双亲委派

Java 虚拟机对 class 文件采用的是按需加载的方式，也就是说当需要使用该类时才会将它的 class 文件加载到内存生成 class 对象，而且，加载某个类的 class 文件时，Java 虚拟机采用的是双亲委派机制，即把请求交由父类处理，它是一种任务委派模式。

通过流程图可以知道双亲委派机制，当要有一个.class 文件需要被加载时，不考虑我们自定义类加载器

  1. 首先会在 AppClassLoader 中检查是否加载过，如果有那就无需再加载了，如果没有则会拿到父类加载器，然后调用父加载器的 loadClass 方法。
  2. 父类中同理也会先检查自己是否已经加载过，如果没有再往上。类似于递归直到 Bootstrap classLoader 之前，都是在检查是否加载过，并不会选择自己去加载。
  3. 直到 BootstrapClassLoader，已经没有父加载器了，这时候开始考虑自己是否能加载了，如果自己无法加载，会下沉到子加载器去加载。
  4. 一直到最底层，如果没有任何加载器能加载，就会抛出 ClassNotFoundException。

# 双亲委派机制的作用

  1. 避免类的重复加载
  2. 保护程序安全，防止核心 API 被随意篡改

# 实验

对于怎么个保护安全程序，这里做几个实验来展示双亲委派的作用

我们可以尝试自己写一个 java.lang.String 类运行，看看能不能加载自定义的 String 类

如果已经看懂双亲委托机制的话，这里显而易见是不能加载的，因为 String 类已经被加载过了，且加载器是 Bootstrap classLoader ，也就是说无论如何自定义类似于 String 这种 java 核心类，都会被 Bootstrap classLoader 先加载 java 自带的，从而无法加载自定义的同名类

通过委托的方式，保证核心.class 不被篡改，即使被篡改也不会被加载，也就是通过双亲委托机制来保护程序安全，防止核心 API 被随意篡改
