# hash 表和 hash 冲突

hash 表也称为散列表，它是基于高速存取的角度设计的，是一种典型的 "空间换时间" 的做法

Hash table 是依据关键码值 (Key value) 而直接进行訪问的数据结构，也就是说，它通过把关键码值映射到表中一个位置来訪问记录，以加快查找的速度。这个映射函数叫做 [散列函数].{red}，存放记录的数组叫做散列表；

散列函数

将元素的关键字 k 通过函数 f 映射到存储地址 p ，即 p=f(k). 创建哈希表时按上述规则将关键字为 k 的元素存到 p=f(k) 位置，查找元素时再利用哈希函数计算出该元素的位置，从而实现按关键字存储元素。
但是哈希表存在冲突的情况，即 p=f(k1)=f(k2) ，但是 k1≠k2 . 实际中冲突不可避免，只能设法减少冲突。

散列函数会影响 hash 冲突的概率；比如我用 key%5 作为散列函数，如果 key = 1, 的到的哈希值是 1，用得到的值建立哈希表，还有其对应的 value 值，如果我 key = 6，得到的哈希值也为 1，这样就产生了冲突。除了散列函数会影响还有其他因素

hash 冲突取决于

  1. 散列函数，一个好的散列函数的值应尽可能分布均匀
  2. 除了冲突的方法
  3. 负载因子的大小。太大不一定好且浪费空间严重，负载因子和散列函数是联动的

常用处理冲突的两种方法

  1. 开放寻址法
  2. 链地地址法 (也叫拉链法)

# HashMap 核心知识

HashMap 是在面试关于 Java 方面最容易被问到的问题；是 HashMap 长得帅吗，让面试官对它情有独钟？

显然不是，这主要还是在于 HashMap 是一个具有代表性的数据存储结构，内部原理和设计思维在很多技术场景都有异曲同工的效果，这个问题是探测你技术基础的扎实情况，一个非常常用的 HashMap 都没有深入学习和了解，可能会认为你在其他技术上也不会有太多深度。

# HashMap 组成结构

HashMap 的结构是由 数组 和 链表 组成

根据上图可以看到，当我们需要从 HashMap 查询一个数据时，首先会找到数组对应下表的头部元素，而这个头部元素就是我们链表的头，再根据链表的头部元素往下一个个匹配直到找到想要的数据，匹配完也没有找到对应数据则返回 null。

为什么 HashMap 要用 数组 + 链表？

因为性能，这就不得不说说 数组 和 链表 的数据结构了

# 数组

因为每个数组都是有下标的，我们可以通过算法做查找优化，所以数组查找数据的速度非常快

但是数组也有缺陷，那就是增删效率会很低，效率和数组长度成反比，数组越长，修改数据下标越靠前，那么增删效率就会越低，这是因为每增加和删除都需要将后面的数据整体向后移动或向前移动。

所以为了弥补数组这个缺陷，用到了另一个数据结构 链表

# 链表

链表的结构不像数组那样会记录下标，但是链表的每个元素自己都会记录下一个元素的位置，虽然查询效率会很慢，但是插入删除效率高，因为插入只需要将插入的节点前后引用修改即可，不需要像数组一样挪动数组。

因为 HashMap 增删改查的操作都比较频繁，所以保证查找效率的同时也保证增删效率就会用到 数组 和 链表 的优势，两种数据结构提升整体性能。

# HashMap 初始化容量机制

当我们进行 put 数据到 HashMap 时，首先要初始化一个数组，然而数组创建是需要指定长度的，当我们没有指定长度时，HashMap 会默认初始化一个长度为 16 的数组。

这里的初始化体现在源码的 resize() 方法，不仅在初始化容量会用到，扩容时也会用到，方法中有一个常量 DEFAULT_INITIAL_CAPACITY 控制 HashMap 初始容量。

# 扩容机制

当数组到达一定长度后，容量不够时，就要对数组进行扩容

源码中，扩容的决定性因素是 DEFAULT_LOAD_FACTOR 负载因子 这个常量上，它的值为 0.75

负载因子设置成 0.75 是经过科学计算的，设置太小会导致 HashMap 空间利用率不高，扩容频率也变高，扩容会将数据重新计算哈希排列，影响性能。

当数据量超过数组容量的 0.75 时，产生 hash 冲突的几率会很高，因为 hash 冲突的数据会放到同一个链表，这样会加长链表的长度，同样也会影响 HashMap 的性能

# 扩容长度

初始化容量为 16，进行扩容时是对原来的容量增加一倍的长度，这里保证的是扩容后的长度是 2 的 N 次幂，为什么保证这种机制，这关于 key 如何被分配到数组的某一位置时结合说明。

# 最大容量

源码中有一个 MAXIMUM_CAPACITY 的常量，该值换算出来为 1073741824，约十亿

这里主要涉及到的知识点就是 HashMap 初始化容量的大小 16，以及这个大小和后面的扩容机制有什么关系，这里主要是为了让 HashMap 的容量大小为 2 的 N 次幂，然后负载因子为什么这么设置，设置成 0.75 是为了减少扩容的频率，更科学合理的利用空间之余又尽量避免产生哈希冲突。

# 定位 key

假设我们得到一个数组，长度为 16，那此时如何定位一个 Key 存储到数组的哪个位置？

因为数组长度是 16，所以有一个硬性要求就是将 Key 的存放位置定位到数组的 0-15 下标之间

  1. 首先会通过 Key.hashCode() 方法得到一个数字，也就是哈希值
  2. 然后会对这个哈希值和数组长度进行一次运算，得到 1-15 的数字，这里先使用 hashCode 对数组长度 16 来进行取余计算得到 1-15 的数字；（注意 HashMap 源码用的是另一种算法，而该算法效率高于取余的算法）
  3. 我们使用取余方式将 hashCode % 16 = n ，最后得到 Key 为”name” 数据会存储到数组下标为 n 的位置，最后我们可以把数据存储到下标为 n 的链表里面去，如果链表里面有相同的 Key 则替换，没有重复的则追加到链表的尾部

这里从源码看出 HashMap 的下标定位算法，在 putVal() 方法中

  • n 为 HashMap 的容量，也就是数组长度

  • i 为此次插入元素的下标

  • (n-1) & hash 将数组长度 - 1，并与 hashCode 进行位与运算

# 位与运算

两个数都转为二进制，然后从高位开始比较，如果两个数都为 1 则为 1，否则为 0，例如

hashCode = 1111111011
size     =       1111
hashCode & size = 0101

再将运算后的值转为十进制则得到元素下标

并且这种算法和取余的方法得出的结果是一致的，且效率更高，但是有一个前提条件，那就是上面提到的扩容机制，保证数组长度为 2 的 N 次幂

# 解决链表过长

HashMap 已经很努力的想办法减少哈希冲突了，但是数组的长度终归是有限的，这样就必然造成一部分数据会分配到同一个数组下标，出现这种情况查询链表效率低。

为了解决这种情况，将链表转为红黑树，通过树结构来优化查询性能。

那么什么时候 HashMap 会触发链表转换成红黑树的操作呢，还有当红黑树的数据少于多少的时候又会转回链表呢，这里我们也可以从 HashMap 源码中找到答案。

当进行 put 数据的时候，链表长度大于 static final int TREEIFY_THRESHOLD = 8;

因为我们的 HashMap 会进行扩容，扩容后我们的元素又可能会分配到别的数组小标里面去，所以我们红黑树的节点也会减少，而当红黑树的节点数量减少到一定程度的时候，红黑树又会转换成链表，而这个值是由 static final int NTREEIFY_THRESHOLD = 6; 来设定的，当红黑树的节点小于等于 6 的时候会完成转换链表的操作，这个我们在 resize () 方法中有调用 split () 方法，从源码中可以看到进行红黑树转链表的操作。

因为数组长度有限的，所以无法避免哈希冲突造成链表数据过多，当我们链表数据过长的时候会进行一些优化，把链表转换为红黑树优化查询效率，当链表长度大于等于 7 的时候，进行链表转红黑树，当链表长度小于等于 6 的时候又会把红黑树还原成链表。

# HashMap 查找对应数据

当明白 put 数据之后，对如何查询、修改和删除数据都很简单。

  1. 计算 key 的哈希值
  2. 通过哈希值定位到 key 存在哪一个数组的下标
  3. 找到后看数组下标是否有节点
  4. 有节点的话，区分节点数据是红黑树还是链表，并且分别使用对应数据结构的查询方法
  5. 根据查找的 key 和 节点里面存的 key 值使用 equas() 判断 hashCode 是否相等，如果 equas 则返回对应的节点，否则继续匹配下一个节点，直到匹配成功返回节点，否则返回 null
