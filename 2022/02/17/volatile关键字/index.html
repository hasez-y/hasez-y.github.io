# volatile 关键字

在 Java 编程中，常遇到并发问题就会想到使用 synchronized 和 volatile 关键字，本篇探讨 volatile 关键字如何解决并发问题的。

并发层面的问题主要包含三个：

  1. CPU 切换指令执行导致的原子性问题
  2. CPU 缓存导致的可见性问题
  3. 编译器和处理器进行指令优化导致的指令重排序问题

代码层面 volatile 是针对指令重排序问题和缓存可见性问题的解决方案

volatile 关键字的两层语义

一旦一个共享变量（类的成员变量、类的静态成员变量）被 volatile 修饰之后，那么就具备了两层语义：

  1. 保证了不同线程对这个变量的操作时的可见性，即一个线程修改该变量的值，该新值对其他线程来说是立即可见的。
  2. 禁止指令重排序

这个代码也许在大多数时候，线程 1 会中断，但也又可能导致无法中断，概率很小，但是发生就会死循环。

每个线程在运行过程中都有自己的工作内存，那么线程 1 在运行的时候，会将 stop 变量的值拷贝一份放在自己的工作内存当中。那么当线程 2 更改了 stop 变量的值之后，但是还没来得及写入主存当中，线程 2 转去做其他事情了，那么线程 1 由于不知道线程 2 对 stop 变量的更改，因此还会一直循环下去。

但是使用 volatile 修饰之后就不一样了：

  1. 使用 volatile 关键字会强制将修改的值立即写入主存
  2. 使用 volatile 关键字，当线程 2 进行修改时，会导致线程 1 的工作内存中缓存的变量 stop 失效
  3. 由于线程 1 的工作内存中缓存变量 stop 的缓存行无效，当需要变量 stop 时就会再次请求主存来拉取最新的 stop 变量值

# 关于原子性

volatile 无法对变量的任何操作都是原子性，就算使用关键字修饰后每次读取的值都是最新的，但是也会出现多个线程同时读取都是最新的，且都修改后写入主存，这就涉及到原子性操作问题了。

使用 synchronized 、 Lock 、 AtomicInteger 关键字可以达成原子性操作。

在 java 1.5 的 java.util.concurrent.atomic 包下提供了一些原子操作类，即对基本数据类型的 自增（加 1 操作），自减（减 1 操作）、以及加法操作（加一个数），减法操作（减一个数）进行了封装，保证这些操作是原子性操作。atomic 是利用 CAS 来实现原子性操作的（Compare And Swap），CAS 实际上是利用处理器提供的 CMPXCHG 指令实现的，而处理器执行 CMPXCHG 指令是一个原子性操作。

# 有序性

在前面提到 volatile 关键字能禁止指令重排序，所以 volatile 能在一定程度上保证有序性。

  1. 当程序执行到 volatile 变量的读操作或写操作时，在其前面的操作的更改肯定全部已经进行，且结果对于后面的操作可见；在其后面的操作肯定还没有进行
  2. 在进行指令优化时，不能将对 volatile 变量的访问语句放在其后面执行，也不能把 volatile 变量后面的语句放到其前面执行

# volatile 的原理和实现机制

前面讲述了源于 volatile 关键字的一些使用，下面我们来探讨一下 volatile 到底如何保证可见性和禁止指令重排序的。

《深入理解 Java 虚拟机》中有描述 "观察加入 volatile 关键字和没有加入 volatile 关键字时所生成的汇编代码发现，加入 volatile 关键字时会多出一个 lock 前缀指令"

lock 前缀指令实际上相当于一个内存屏障，也叫内存栅栏，内存屏障会提供 3 个功能：

  1. 它确保指令重排序时不会把其后面的指令排到内存屏障之前的位置，也不会把前面的指令排到内存屏障的后面；理解为执行到内存屏障这句指令时，在它之前的操作已经全部完成。
  2. 它会强制将对缓存的修改操作立刻写入主存
  3. 如果是写操作，它会导致其他 CPU 中相应的缓存行无效

# 使用场景

synchronized 关键字是防止多个线程同时执行一段代码，那么就会很影响程序执行效率，而 volatile 关键字在某些情况下性能要优于 synchronized ，但是要注意 volatile 关键字是无法替代 synchronized 关键字的，因为 volatile 关键字无法保证操作的原子性。
