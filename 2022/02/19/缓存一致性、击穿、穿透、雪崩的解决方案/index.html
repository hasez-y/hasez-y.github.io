# 缓存一致性

一般情况缓存都只是为了提高数据检索性能的一个媒介，它不会作为最后的数据存储归宿，最终数据都会存储到数据库，也就是硬盘中。

但是会出现一个问题，像数据库数据和缓存的数据不同步，这就是缓存一致性的问题。

解决方案：

  1. 先删除缓存，再修改数据库

    导致问题主要原因是修改缓存这个操作存在并发问题，那么我们去掉修改这个操作即可，每次修改都会删掉缓存相应的数据，当查询时，发现缓存没有则会向数据库查询，查询出来后再 insert 缓存中。这方案也有概率会因为并发翻车....

  2. 更新缓存时加锁

  3. 设置较短的过期时间

    设置较短的过期时间，只要数据一过期，程序就会从数据库同步到缓存，这种方案虽然不能直接解决缓存不一致的问题，但是可以控制不一致的数据存在时长

我觉得可以将 1、3 方案结合起来减少缓存不一致的问题。

# 缓存击穿

缓存本身就是存储一些经常使用的活跃数据，对于一些不常用的数据不必要长久的保存在内存中，因为内存资源有限，所以缓存数据会设置一定的失效时间，缓存失效后，可以再从数据库读取并同步到缓存中。

而缓存击穿出现在数据过期这个场景中，缓存过期后，那么查询数据的请求不得不全打在数据库上，如果并发量顿时非常大，就像双十一购物一样，热点缓存数据又恰好失效，全部打在数据库，又可能会因为数据库承受不住而崩溃。

解决方案：

缓存击穿的核心是热点数据同时过期，那么只要避免数据过期即可

  1. 当从缓存数据过期，重新从数据库加载数据到缓存的过程上互斥锁
  2. 把过期时间控制在低流量的时间段
  3. 不设置过期时间，这个可以避免热点数据失效问题，但通常不建议

# 缓存穿透

缓存穿透通常是说，请求的数据再缓存里面不存在，同时数据库也不存在该数据；因为系统里根本不存在该数据，所以请求总是无法命中缓存，每次都会进入数据库，导致数据库承受不住大量请求导致崩溃。

# 恶意穿透

明知搜索的数据根本不存在还不断请求，属于恶意攻击，这种恶意请求通常有目的性的频繁随机请求系统不存在的数据，造成请求穿透缓存打在数据库上，导致数据库瘫痪风险。

解决方案：

  1. 缓存 null 值

    如果数据不存在则缓存 null，每次请求该数据时就会命中缓存。但是恶意攻击通常随机参数请求，缓存 null 会造成缓存大量数据，造成内存风险。

  2. 布隆过滤器

    说起布隆过滤器前，先说一种方案，就是将数据库所有数据同步到缓存中，请求进来时发现缓存没有则直接返回，这样就不会进入数据库了，但是问题也很明显，数据量一大就会非常浪费内存。

    而布隆过滤器就解决上面浪费内存的问题，布隆过滤器是一种数据结构，比较巧妙的概率型数据结构（probabilistic data structure），特点是高效地插入和查询，可以用来告诉你 “某样东西一定不存在或者可能存在”；

    简单说布隆过滤器是通过位图来记录一个数据通过不同哈希函数生成的多个哈希值，并对生成的哈希值指向的 bit 位置标为 1。当请求的数据我们想知道系统中到底存不存在时，就会重复以上步骤得到若干哈希值对应到位图上，如果位图上相应 bit 位都是 1，那么该数据库可能存在该数据，但是只要有一个 bit 位是 0，那么必定不存在该数据。这样就完成了节省内存的同时过滤部分恶意的请求了，具体了解 : https://zhuanlan.zhihu.com/p/43263751

# 缓存雪崩

如在高并发的情况下，缓存同一时刻失效（如缓存挂了，或者设置了相同过期时间），所有请求会读数据库，容易导致数据库负载瞬间上升，乃至崩掉。如果重启数据库，立马又会被新的请求压崩。

解决方案:

  1. 缓存的失效时间设置为随机值，避免同时失效
  2. redis 搭建高可用，主从 + 哨兵，redis cluster
  3. 服务限流、降级，避免数据库被瞬间压崩

# 缓存击穿和缓存雪崩区别

两者有点像，主要区别在于缓存击穿是查询同一条数据或热点数据查询不到，穿过了缓存。缓存雪崩是指大量数据查询不到，穿过了缓存。

# 系统保护机制

# 限流

限流是通过事先预防的机制来保护系统，我们通过事先对系统可承受的流量进行估算，从而设定一个阈值，超过这个阈值后的请求 不再向下层发送，而是直接拒绝超过阈值的请求。

# 熔断

限流是事先预防，而熔断更多的是事后保护机制，当某些功能出现延时或者故障后，系统会触发熔断机制，直接拒绝处理请求，从而故障进一步的扩散，引发雪崩效应。

# 降机

当系统流量太大，超出系统负荷时，避免大负荷量拖垮我们整个系统，这个时候我们就有必要对业务的功能进行取舍了，为了保证核心业务能正常运转，从而暂时关闭一些非核心的业务，根据我们的系统业务等级划分来保护关键业务的正常运行，比如说一个电商网站 关键业务就是选购商品、下单、支付，至于其他的功能资讯、活动、消息、个人信息都可以先关闭，从而保证系统能进行最基础的购物功能。
