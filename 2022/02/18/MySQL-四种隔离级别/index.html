# MySQL 隔离级别

前面介绍了 MySQL 事务的 ACID 原则，也就是原子性、一致性、隔离性、持久性，其中事务的隔离性是指当多个事务同时运行时，各个事务互相隔离，不可互相干扰。

如果事务没有隔离性，就会出现脏读、不可重复读、幻读等情况

  1. 脏读：脏读又称无效数据的读出，是指在数据库访问中，事务 T1 将某一值修改，然后事务 T2 读取该值，此后 T1 因为某种原因撤销对该值的修改，这就导致了 T2 所读取到的数据是无效的，值得注意的是，脏读一般是针对于 update 操作的
  2. 不可重复读：是指在一个事务内，多次读取同一个数据；假设一个事务还没结束，另外一个事务也访问了该同一个数据，那么第一个事务中的两次读数据之间，由于第二个事务的修改，导致第一个事务两次读取同一个数据是不一样的；读取到两次数据不一样，称之为不可重复读
  3. 幻读：是指当事务不是独立执行时发生的一种现象，例如第一个事务对一张表的修改涉及到全部数据行，而碰巧另一个事务也修改了该表数据，向该表插入了一行新数据，那么就会发生操作第一个事务的用户发现表中还有没有修改的数据行，这就好像发生了幻觉

为了解决以上问题，标准 SQL 定义了 4 类事务隔离级别，来指定事务中的哪些数据该表是可见的，哪些数据改变是不可见的。

  1. 读未提交 (Read Uncommitted)
  2. 读提交 (Read Committed)
  3. 可重复读 (Repeatable Read)
  4. 串行化 (Serializable)

各个事务隔离级别会产生问题如下表：

# 读未提交 Read Uncommitted RU

如果一个事务读取到了另一个未提交事务修改过的数据，那么这种隔离级别称之为读未提交

在该隔离级别下，所有事务都可以看到其它未提交事务的执行结果。因为它的性能与其他隔离级别相比没有高多少，所以一般情况下，该隔离级别在实际应用中很少使用。

# 读提交 Read Committed RC

如果一个事务只能读取到另一个已提交事务修改过的数据，且其他事务每对该数据进行一次修改并提交后，该事务都能查询得到最新值，那么这种隔离级别称之为读提交

该隔离级别满足了隔离的简单定义：一个事务从开始到提交前所做的任何改变都是不可见的，事务只能读取到已经提交的事务所做的改变。

这是大多数数据库系统的默认事务隔离级别，例如 Oracle、SQL Server，但 MySQL 的 InnoDB 引擎默认的事务隔离级别不是

使用这种隔离级别时，会出现不可重复读现象。

# 可重复读 Repeatable Read RR

在一些场景，一个事务只能读取到另一个已提交的事务修改过的数据，但是第一次读过的某条数据后，即使其他事务修改了该数据的值且提交，之后改事务再读该记录时，读到的仍是第一次读到的值，而不是读取到最新的值，这种隔离级别称之为可重复读

可重复读是 MySQL InnoDB 默认事务隔离级别，它确保同一事务的多个实例再并发读取数据时，会看到同样的数据行。在该隔离级别下，如果有事务正在读取数据，就不允许有其他事务进行修改操作，这样就解决了可重复读问题。

但是会产生幻读问题，例如 A 窗口开启事务查询 id 为 1 的记录时，结果为空，正要插入一条 id 为 1 的记录时，此时另外一个窗口 B 提交了一个事务，事务就是插入一条 id 为 1 的记录，此时 A 窗口插入时就会插入失败，但是查询 id 为 1 的记录时还是为空，这就是幻读问题。

# 串行化 Serializable

如果一个事务先根据某些条件查询出一些记录，之后另一个事务又向表中插入了符合这些条件的记录，原先的事务再次按照该条件查询时，能把另一个事务插入的记录也读出来。那么这种隔离级别就称之为串行化

SERIALIZABLE 是最高的事务隔离级别，主要通过强制事务排序来解决幻读问题。简单来说，就是在每个读取的数据行上加上共享锁实现，这样就避免了脏读、不可重复读和幻读等问题。但是该事务隔离级别执行效率低下，且性能开销也最大，所以一般情况下不推荐使用。
